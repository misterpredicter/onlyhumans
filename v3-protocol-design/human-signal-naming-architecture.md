## Part 1: Name Exploration

### The Naming Problem

The product does three things that normally get three different names:

1. **Infrastructure** — a protocol/API where agents buy verified human judgment
2. **Consumer** — a verified-human social network where tasks are content
3. **Financial** — judgment markets where subjective questions get priced

The name needs to work for a hackathon demo on Sunday AND a Series A deck in 18 months. It needs to feel like a protocol (developers), a product (workers), and a market (investors) depending on who's reading it.

## Part 2: Brand Architecture Options

### Option A: One Brand Rules All

**Pick one name. Use it for everything.**

Example: **Human Signal** (or whichever name wins)

| Context | How It Shows Up |
|---------|-----------------|
| Website | humansignal.xyz — "Human judgment as infrastructure" |
| API docs | docs.humansignal.xyz — "Human Signal API Reference" |
| Social | @humansignal everywhere |
| Pitch to investors | "Human Signal is a judgment market protocol" |
| Pitch to developers | "pip install humansignal / npm install @humansignal/sdk" |
| Pitch to workers | "Earn USDC for your judgment on Human Signal" |
| Social network | "Human Signal" (the app) |
| Judgment markets | "Human Signal Markets" (a feature, not a brand) |
| Data licensing | "Human Signal Data" (a revenue stream, not a brand) |

**Pros:**
- Simple. One name, one brand, one story.
- All marketing spend compounds into one identity.
- "Human Signal" works because "signal" scales — it's a signal feed, a market signal, a social signal.
- Hackathon-friendly: no brand architecture to explain.

**Cons:**
- The social network wearing the same name as the API feels weird. Instagram isn't called "Meta."
- If the consumer app takes off and the protocol fades (or vice versa), the brand is trapped in the wrong connotation.
- "Human Signal" has a B2B protocol energy that won't attract Gen Z workers to a social app.

**Verdict:** This is the correct choice for the hackathon and probably for the next 12 months. Brand architecture is a scaling problem, not a launch problem.

---

### Option B: Protocol + Products

**One name for the protocol layer. Different names for each product direction.**

Example:
- **Human Signal Protocol** — the open infrastructure layer (API, SDK, standards)
- **OnlyHumans** — the consumer social app where workers judge and earn
- **Signal Markets** — the judgment market / prediction market product
- **Signal Data** — the licensed dataset business

| Context | How It Shows Up |
|---------|-----------------|
| Website | humansignal.xyz — protocol landing page + links to products |
| API docs | docs.humansignal.xyz — developer-facing |
| Social | @humansignal (protocol), @onlyhumans (consumer app), @signalmarkets (markets) |
| Pitch to investors | "Human Signal Protocol powers three products: OnlyHumans, Signal Markets, Signal Data" |
| Pitch to developers | "Build on Human Signal Protocol" |
| Pitch to workers | "Join OnlyHumans — get paid for your judgment" |
| Judgment markets | signalmarkets.xyz |
| Data licensing | "Signal Data — the verified human judgment corpus" |

**Pros:**
- Each product can have its own voice and audience. OnlyHumans can be provocative and consumer-y. Signal Markets can be serious and financial. The protocol can be neutral and developer-friendly.
- The Stripe model: Stripe is the protocol, Link is the consumer product, Atlas is the business product. Same company, different brands for different audiences.
- Investors love "platform + multiple product lines" narrative.

**Cons:**
- Three brands is three times the marketing spend.
- Confusing at hackathon stage — "wait, are you Human Signal or OnlyHumans?"
- OnlyHumans carries real brand risk — the OnlyFans parallel could alienate enterprise customers, AI labs, and institutional investors. The name is sticky precisely because it's provocative, and provocative cuts both ways.
- Premature optimization. You don't know which product direction will win yet.

**Verdict:** This is the right architecture at Series A scale IF the consumer app becomes real. Not for the hackathon. The risk with OnlyHumans specifically is that it boxes you into the consumer angle when the protocol and enterprise angle may be where the money is.

### Option C: Umbrella + Sub-brands

**A parent brand that contains everything, with descriptive sub-brands.**

Example:
- **Human Signal** — the umbrella
- **Human Signal Oracle** — the API/protocol
- **Human Signal Social** — the verified-human network
- **Human Signal Markets** — judgment markets
- **Human Signal Data** — licensed datasets

| Context | How It Shows Up |
|---------|-----------------|
| Website | humansignal.xyz — unified, with tabs/sections for each product |
| API docs | "Human Signal Oracle API" |
| Social | @humansignal everywhere (one account, one voice) |
| Pitch to investors | "Human Signal has four product lines under one brand" |
| Pitch to developers | "Integrate Human Signal Oracle into your agent pipeline" |
| Pitch to workers | "Join Human Signal — the verified-human judgment network" |

**Pros:**
- Simplest to execute. No new names to register, no brand confusion.
- Every product benefits from the parent brand's growing recognition.
- Easy to add new product lines without new branding exercises.

**Cons:**
- "Human Signal Social" is a terrible consumer brand. No one downloads an app called "[Protocol Name] Social."
- Descriptive sub-brands are forgettable. They're organizational labels, not brands.
- The umbrella constrains all products to the same tone and audience, which is the exact problem you're trying to solve.

**Verdict:** This is the enterprise play. It works if the primary audience is AI labs and developers, and the consumer app is a secondary growth vector. It does NOT work if the consumer app needs to stand on its own.

---

### Recommended Architecture

**For the hackathon (this weekend):** Option A. Everything is "Human Signal." Period. Don't spend one second on brand architecture. The name works, the product is live, ship it.

**For post-hackathon (next 3 months):** Stay with Option A. See which direction gets traction. If agents adopt the API, the protocol name matters most. If workers love the social experience, a consumer brand will emerge organically. Don't name what you haven't built.

**For Series A (12-18 months):** Option B, but only if the consumer app becomes real. Keep "Human Signal" for the protocol/company. Give the consumer app its own name — but pick that name based on what the community actually calls it, not what sounds good in a doc. OnlyHumans is one option. The right answer will be obvious by then because you'll have users.

**The rule:** Name things after they exist, not before.

## Part 3: Positioning Statements

### For "Human Signal" (the protocol + hackathon name)

**One-sentence pitch to AI developers/agents:**
> Human judgment as an API — post a question, pay via x402, get verified human answers in 60 seconds with World ID sybil resistance.

**One-sentence pitch to human workers/participants:**
> Get paid in USDC for your taste, judgment, and expertise — every response verified by World ID, every payment instant.

**One-sentence pitch to investors:**
> The marketplace where AI agents buy the one thing they can't produce — verified human cognition — settling $4B/year in demand through a supply chain that's cryptographically sybil-proof for the first time.

**One-sentence pitch to hackathon judges:**
> The first product to compose World ID, x402, and XMTP into a single protocol — where AI agents pay verified humans for judgment, settled per-response on Base.

**The "X for Y" formulas:**
- "Twilio for human judgment" — (best for developers; positions as infrastructure/API)
- "Chainlink for subjective questions" — (best for crypto-native audience; oracle positioning)
- "The sybil-proof Scale AI" — (best for AI lab buyers; direct competitive positioning)
- "Polymarket for taste" — (best for the judgment markets angle; but narrows scope)

**Recommended default:** "Twilio for human judgment." It's the most scalable framing. Twilio didn't limit itself to voice or SMS — it was "communications infrastructure." Human Signal shouldn't limit itself to RLHF or A/B testing — it's "judgment infrastructure."

## Part 4: The Meta Question — Is "Human Signal" Actually Right?

### The Case For Keeping It

**"Signal" is the right word.** The product produces a signal — a real-time, priced, verified data stream of what humans think. Not a report. Not a batch of labels. A signal. That word works for the API (agents consume the signal), for markets (the market price IS the signal), and for the data business (the dataset is a corpus of signals). The V3 spec's own conclusion was that "signal" is better for the market framing than for the survey framing — and the market framing is the one you're moving toward.

**"Human" is the right qualifier.** In a world of AI-generated everything, the differentiator is provably human origin. "Human Signal" says exactly what it is: the signal that comes from humans. Not bots. Not LLMs completing your annotation tasks. Humans.

**The name has momentum.** It's live at themo.live. The README uses it. The pitch deck uses it. The codebase is named it. Renaming costs time and creates confusion, and you're 48 hours from a hackathon deadline.

**Protocol names should be boring.** HTTP, TCP/IP, SMTP, x402 — protocols have descriptive names, not clever ones. "Human Signal" is descriptive. That's a feature for the protocol layer.

### The Case Against

**It's generic.** Search "human signal" and you get psychology papers, HR consulting firms, and a podcast. There's zero distinctiveness. If this becomes a real company, you'll spend years and dollars establishing brand recognition against noise.

**It doesn't capture the market mechanic.** "Signal" is passive — you receive a signal. But the judgment market is active — people trade, stake, price, resolve. "Human Signal" sounds like a data feed, not an exchange. If judgment markets become the core product (the V3 spec suggests they should), the name undersells the mechanism.

**It doesn't capture the social layer.** "Human Signal" sounds like B2B infrastructure. It does not sound like a place where verified humans scroll, vote, earn, and socialize. If the consumer flywheel matters (and the V3 spec argues it's critical for cold start), the name creates a branding ceiling.

**There's a direct collision.** HumanSignal is a company — it makes Label Studio, an open-source data labeling tool. Same space, same keywords. This creates SEO competition and brand confusion from day one.

### The Honest Answer

**Human Signal is the right name for right now.** It's accurate, it's live, and it doesn't limit what you can build next. The name scales from "API for verified human judgment" to "judgment market protocol" without breaking. The places where it doesn't work (consumer social, provocative branding) are problems you don't have yet.

**The name will probably need to change at Series A.** Not because it's wrong, but because:
1. The HumanSignal/Label Studio collision will become a real problem as you scale
2. A Series A company needs a name that's distinctive enough to own in search, social, and press
3. By then you'll know which product direction won, and you can name accordingly

**What to do this weekend:** Ship as Human Signal. Don't think about naming for one more second.

**What to do in 3 months:** If the protocol gets traction, register a distinctive domain and start migrating. If the consumer app gets traction, name it separately. If judgment markets get traction, consider whether "Signal Markets" or something bolder captures it better.

**The name that would work across all three directions if you HAD to pick one today:** Honestly, the best candidate that doesn't already exist as a company is something in the **Demos** family. "Demos" (Greek: "the people") captures the democratic judgment mechanic, scales from protocol ("Demos Protocol") to consumer ("Demos" the app) to markets ("Demos Markets"), sounds distinctive, and connotes both governance and collective intelligence. demos.market or demos.xyz could work as domains. But don't do this now. Do this when you have traction and know which direction is winning.
